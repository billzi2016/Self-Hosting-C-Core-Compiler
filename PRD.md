# C-Core-Compiler PRD

## 1. 项目概述

### 1.1 项目名称

`C-Core-Compiler`

### 1.2 项目定位

这是一个使用 Python 从零实现的教学型与工程化编译器项目，目标是将一个经过约束的 C 语言子集编译为可在 macOS 或 Linux 上直接运行的原生可执行程序。

项目强调以下几点：

- 编译原理理解扎实，而不是只会调用现成工具链
- 工程结构清晰，代码可读性强，注释完整
- 测试覆盖全面，能够证明编译结果正确
- 最终产物真实可运行，能够体现端到端实现能力

### 1.3 项目目标

本项目的核心目标不是实现完整 C 编译器，而是实现一个“范围可控、结构完整、结果可信、便于展示”的 C 子集编译器。该编译器应具备从源代码输入到可执行文件输出的完整链路，包括：

- 词法分析
- 语法分析
- 抽象语法树构建
- 语义分析
- 中间表示生成
- 基础优化
- 目标代码生成
- 调用系统汇编与链接工具生成可执行文件

### 1.4 项目价值

该项目应清晰体现以下工程特征：

- 编译流程完整且逻辑清晰
- 能独立设计复杂系统的模块边界
- 能将理论落地为可执行工程
- 能兼顾正确性、可维护性与测试完备性
- 能处理跨平台输出差异

## 2. 产品愿景

项目最终应呈现以下特征：

- 这不是脚本拼接，而是一个结构完整的编译器
- 从源代码到机器执行之间的关键步骤划分清楚
- 使用 Python 构建复杂工程时仍保持良好的结构化设计
- 项目重视质量，尤其重视测试、错误信息、可读性和注释
- 项目逻辑清晰，文档、模块和数据流都容易追踪

## 3. 范围定义

### 3.1 一期目标

一期实现一个可用的 C 子集，支持从 `.c` 文件编译到本机可执行文件。

建议一期支持：

- 整型类型：`int`
- 局部变量
- 全局变量
- 函数定义
- 函数调用
- 参数传递
- 返回值
- 块语句
- 表达式语句
- 赋值语句
- `if / else`
- `while`
- `for`
- `return`
- 一元运算：`+ - !`
- 二元运算：`+ - * / %`
- 比较运算：`== != < <= > >=`
- 逻辑运算：`&& ||`
- 括号改变优先级

### 3.2 明确不做

为保证项目聚焦，一期不支持以下能力：

- `float`、`double`
- 指针算术的完整语义
- 结构体、联合体、枚举
- 预处理器完整实现
- 宏展开
- 多文件编译与链接优化
- 完整标准库兼容
- 复杂 ABI 兼容
- 自举编译

### 3.3 二期预留

二期可根据时间继续扩展：

- 指针与数组
- 字符串字面量
- `char`
- 递归优化
- 更完整的作用域系统
- 简单寄存器分配优化
- 常量折叠与死代码删除增强
- 输出中间表示可视化

### 4.2 关键场景

## 5. 核心需求

### 5.1 输入输出需求

输入：

- 单个 `.c` 源文件

输出：

- 目标平台汇编文件
- 可选中间表示文件
- 可执行文件
- 编译错误信息

命令形式建议：

```bash
python -m c_core_compiler input.c -o out
```

可选参数建议：

```bash
--emit-tokens
--emit-ast
--emit-ir
--emit-asm
--target linux-x86_64
--target macos-x86_64
--run
```

### 5.2 可执行文件需求

项目必须满足以下结果要求：

- 在 Linux 上可生成并运行 Linux 可执行文件
- 在 macOS 上可生成并运行 macOS 可执行文件
- 不要求单机交叉编译
- 优先要求“在对应平台本机可编译可运行”

这意味着技术路线应采用：

- 前端统一：词法、语法、AST、语义、IR 全平台共用
- 后端分离：针对 Linux 和 macOS 生成不同格式的目标汇编
- 最终通过系统现成工具链完成汇编和链接

### 5.3 平台策略

不依赖 NASM 作为核心前提。为了兼容 macOS，推荐直接生成能被 `clang` 接收的汇编文本，并通过系统工具链产出最终可执行文件。

推荐策略：

- Linux：生成 x86_64 SysV ABI 对应汇编，调用 `clang` 或 `cc` 完成汇编链接
- macOS：生成 x86_64 Mach-O 兼容汇编，调用 `clang` 完成汇编链接

这样做的原因：

- macOS 默认工具链对 `clang` 支持最好
- 避免把项目绑死在单一汇编器语法上
- 便于后续扩展到 ARM64 或更多平台

## 6. 技术路线

### 6.1 语言与依赖约束

项目主体必须使用 Python 实现，并尽量只依赖标准库。

建议使用的标准库：

- `re`
- `dataclasses`
- `enum`
- `typing`
- `pathlib`
- `subprocess`
- `tempfile`
- `textwrap`
- `json`
- `unittest`
- `difflib`
- `heapq`

原则：

- 不使用第三方解析器生成器
- 不使用第三方测试框架作为硬依赖
- 不使用大型编译基础设施作为运行前提

### 6.2 编译架构

推荐采用如下分层：

1. Lexer
2. Parser
3. AST
4. Semantic Analyzer
5. IR Builder
6. Optimizer
7. Code Generator
8. Toolchain Driver

### 6.3 解析策略

推荐手写递归下降语法分析器。

原因：

- 结构清晰，便于理解与维护
- 便于写丰富注释
- 便于逐步调试和测试
- 不依赖外部工具

表达式解析推荐使用：

- Pratt Parser，或
- 分层优先级递归下降

优先建议使用分层优先级递归下降，因为其控制流和优先级处理方式更直观，便于阅读与调试。

### 6.4 中间表示

建议设计一个简洁、自定义、可打印的三地址码风格 IR。

示例：

```text
t1 = const 1
t2 = const 2
t3 = add t1, t2
store x, t3
ret t3
```

IR 的目标：

- 隔离前端与后端
- 提高调试可观测性
- 便于写优化和测试
- 便于展示“编译器不是直接拼汇编”

### 6.5 优化策略

一期只做基础优化：

- 常量折叠
- 常量传播
- 死代码删除
- 基本块内简单代数化简

`heapq` 可用于后续工作列表调度、优化队列或简化某些分析流程，但不是强制要求。

## 7. 代码风格与注释要求

### 7.1 注释要求

项目必须配备高质量注释，且注释应以“解释设计意图和编译行为”为主，而不是解释显而易见的语法。

注释应覆盖：

- Token 规则为什么这样设计
- 每个 AST 节点代表什么语义

- 类型检查的边界条件
- IR 指令为何这样抽象
- 每种控制流如何翻译
- 平台 ABI 差异如何处理
- 为什么这样组织测试

### 7.2 文档要求

至少需要以下文档：

- `README.md`：项目简介、演示方式、亮点、快速开始
- `PRD.md`：项目定义与范围

## 8. 测试策略

### 8.1 总原则

这个项目必须把“正确性与可验证性”作为核心特征之一。

虽然开发过程不强制测试先行，但最终必须形成“测试非常丰富”的工程结果。测试要足够多，足够细，足够自动化，能说明以下事实：

- 词法结果正确
- 语法树结构正确
- 语义错误能被拦截
- IR 生成稳定
- 汇编输出符合预期
- 生成的二进制在真实平台上能运行并得到正确结果

### 8.2 测试分层

必须覆盖以下层次：

1. 词法测试
2. 语法测试
3. AST 快照测试
4. 语义分析测试
5. IR 生成测试
6. 优化器测试
7. 代码生成测试
8. 端到端编译运行测试
9. 错误信息测试
10. 回归测试

### 8.3 测试内容要求

#### 词法测试

- 关键字识别
- 标识符识别
- 整数字面量识别
- 运算符与分隔符识别
- 空白与注释跳过
- 非法字符报错

#### 语法测试

- 表达式优先级

- 变量声明
- 赋值语句
- 条件语句
- 循环语句

#### 语义测试

## 11. 关键设计决策

### 11.1 为什么选择 Python

- 开发速度快
- 适合表达编译器中的树结构与转换流程
- 更容易写清楚注释与调试输出
- 适合用清晰代码表达工程设计与编译流程

### 11.2 为什么只做 C 子集

- 完整 C 语言过于庞大
- 更容易在有限时间内做到真正可用
- 更容易形成高质量测试闭环
- 更容易打磨成结构完整、逻辑清晰的公开项目

### 11.3 为什么要有 IR

- 更像一个真正的编译器
- 易于测试与调试
- 易于扩展优化
- 易于前后端解耦

### 11.4 为什么不把项目绑定到 NASM

- macOS 环境下默认体验并不理想
- `clang` 作为系统工具链接入更自然
- 有利于同时支持 Linux 与 macOS

## 14. 风险与应对

### 风险 1：一开始语法范围过大

应对：

- 严格控制一期只做 C 子集
- 每完成一类语法就补齐测试

### 风险 2：平台后端差异拖慢进度

应对：

- 先统一前端和 IR
- 先打通 Linux 或 macOS 其中一个平台
- 平台特定逻辑收敛在独立后端模块

### 风险 3：没有测试导致后续频繁回归

应对：

- 每个阶段都补齐模块测试
- 所有 bug 都沉淀为固定回归用例

### 风险 4：项目看起来像学习代码，不像工程作品

应对：

- 强化文档与注释
- 强化模块边界与分层设计
- 强化 CLI、错误信息和目录结构
- 强化端到端 demo 与测试质量

## 15. 成功定义

如果最终达到以下状态，则项目成功：

- 读者能在几分钟内理解项目目标与整体结构

- 能运行示例并得到真实可执行文件
- 能通过测试与文档确认项目质量
