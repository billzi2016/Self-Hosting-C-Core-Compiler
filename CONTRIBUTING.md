# Contributing

## 目标

这个项目优先追求三件事：

- 逻辑清晰
- 行为可验证
- 改动可维护

如果提交的新功能会明显拉高复杂度，但不能同时提升可读性、测试覆盖或结构稳定性，优先拒绝这种改动方式。

## 提交原则

请尽量把改动拆成小而清晰的提交：

- 一个提交只解决一个主要问题
- 提交信息尽量直接描述改动结果
- 新功能应附带测试
- 修 bug 应附带回归测试

## 代码风格

- 优先写直白代码，而不是炫技代码
- 注释重点解释“为什么这样设计”，不是重复代码字面意思
- 新文件开头应写清楚这个文件负责什么
- 新增模块要有明确边界，不要把多个阶段逻辑混在一起

## 测试要求

修改前请确认：

- 词法、语法、语义、IR、优化、CLI 或端到端中，受影响层次至少有一层测试覆盖
- 如果引入新语言特性，应至少补一条解析测试和一条端到端测试

建议运行：

```bash
python3 -B -m unittest discover -s tests -v
```

## 适合提交的方向

- 新语言特性，但要有清晰边界
- 更好的错误信息
- 更稳的测试和回归用例
- 更清楚的文档和示例
- 保守、可解释、可验证的优化

## 暂不鼓励的方向

- 一次性大改整个前端或后端
- 没有测试支撑的激进优化
- 把简单逻辑封成难以追踪的抽象层
- 让代码“看起来高级”但更难读的重构
