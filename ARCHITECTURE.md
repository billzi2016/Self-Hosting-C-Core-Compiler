# Architecture

## 第一代总体原则

第一代实现遵循三个原则：

- 逻辑清晰：每个阶段只做自己该做的事
- 结果可验证：每个阶段都能独立输出并测试
- 结构可演进：后续可以在不推倒前端的情况下替换后端或增加优化

## 模块划分

### `tokens.py`

定义 Token 类型、Token 数据结构和关键字映射。

### `lexer.py`

负责把源代码拆成 Token 列表，并补齐行列号信息。

### `ast_nodes.py`

定义 AST 节点，用统一的数据结构表达语法树。

### `parser.py`

负责把 Token 序列解析为 AST。第一代使用手写递归下降，重点是控制流清晰、错误位置可定位。

### `symbol_table.py`

负责管理作用域和符号查找。语义分析阶段用它维护变量和函数定义。

### `semantic.py`

负责做第一代需要的语义检查，例如重复定义、未定义符号、参数数量不匹配和非法赋值目标。

### `ir.py`

定义一种简洁的三地址风格 IR，用来隔离前端和后端。

### `ir_builder.py`

把 AST 逐步翻译成 IR。控制流语句在这里被显式展开成标签与跳转。

### `optimizer.py`

第一代只提供优化管线占位和最小接口，方便第二阶段在不破坏整体结构的前提下加入适度优化。

### `codegen/`

负责将 IR 生成后端代码。第一代后端不是原生汇编，而是规范化 C 代码，目的是优先打通跨平台可执行物和清晰的数据流。

### `toolchain.py`

负责选择后端、写出中间产物、调用 `clang` 或 `cc` 并生成最终可执行文件。

### `cli.py`

负责命令行入口，把各阶段串起来，并提供 `--emit-*` 调试能力。

## 数据流

```text
source
  -> lexer
  -> tokens
  -> parser
  -> AST
  -> semantic analyzer
  -> IR builder
  -> IR
  -> backend code generator
  -> generated C
  -> system toolchain
  -> executable
```

## 为什么第一代后端先输出 C

第一代最重要的目标是完整性、清晰度和可测试性，而不是立刻进入复杂的机器相关细节。输出规范化 C 有这些好处：

- 能快速生成真实可执行文件
- 能在 macOS 和 Linux 上复用系统工具链
- 便于调试每个 IR 指令对应的结果
- 便于后续把后端替换为原生汇编而不动前端

## 错误处理策略

- Lexer 报告非法字符与位置
- Parser 报告当前期望的语法成分
- Semantic Analyzer 报告符号与语义错误
- Toolchain Driver 报告外部工具链失败信息

每一层都尽量在自己的职责边界内报错，避免把错误推迟到更后面的阶段。
